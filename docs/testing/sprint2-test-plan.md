## 1. Introduction

### 1.1 Purpose

This test plan defines the testing strategy, scope, approach, and acceptance criteria for Sprint 2 of the MedVix project. Sprint 2 delivers the first three steps of the 7-step ML pipeline: Step 1 (Clinical Context), Step 2 (Data Exploration), and Step 3 (Data Preparation). The goal of this plan is to ensure that all twelve user stories completed in Sprint 2 meet their acceptance criteria before the sprint gate review on 18 March 2026.

### 1.2 Background

MedVix is an educational ML visualization tool built for healthcare professionals at Cankaya University's SENG 430 Software Quality Assurance Lab. The system allows clinicians to upload patient datasets, explore data quality, apply preprocessing transformations, and ultimately train and interpret ML models within a guided 7-step pipeline.

Sprint 2 builds on the Sprint 1 foundation and adds the core data exploration and preparation capabilities. These features are the gateway to all subsequent ML functionality — a defect in data validation or the column mapper gate will cascade into every downstream sprint.

### 1.3 Document Scope

This plan covers all functional testing for Steps 1-3, including:

- Manual functional testing of the React frontend (localhost:5173)
- Automated pytest testing of the FastAPI backend (localhost:8000)
- Integration testing of frontend-backend communication via the REST API
- Boundary value analysis for file upload and split-ratio constraints
- Negative testing for all validation paths

## 3. Test Approach

### 3.1 Testing Levels

**Level 1 — Unit Tests (Backend)**
- Framework: pytest with pytest-asyncio
- Location: `backend/tests/`
- Coverage target: All service functions in `data_service.py`, `preparation_service.py`, `domain_service.py`
- Runner: `pytest backend/tests/ -v`

**Level 2 — API Integration Tests**
- Framework: pytest with httpx async client via FastAPI `TestClient`
- Target: All 10 REST endpoints registered in Sprint 2
- Approach: Each test sends an HTTP request and asserts status code, response schema, and key field values
- Data: Synthetic CSV fixtures generated by `backend/data/generate_synthetic.py`

**Level 3 — Manual Functional Tests**
- Performed by: Arzu Tugce KOCA
- Tools: Chrome DevTools, Postman, browser console
- Documentation: Sprint 2 Test Cases document (sprint2-test-cases.md)
- Evidence: Screenshots recorded in Jira test issues

### 3.2 Test Execution Order

Tests must be executed in pipeline order to respect session state dependencies:

1. Backend health and domain list endpoints (no session)
2. File upload and built-in dataset loading (creates session)
3. Data summary and preview (requires session with data)
4. Column mapper (requires session with data)
5. Data preparation (requires session with validated column mapping)
6. Preparation status check (requires prepared session)

### 3.3 Test Data Strategy

**Valid CSV Files**:
- `heart_valid.csv` — 303 rows, 13 features, binary target `target` (0/1)
- `breast_valid.csv` — 569 rows, 30 features, binary target `diagnosis` (M/B)
- `diabetes_valid.csv` — 768 rows, 8 features, binary target `Outcome` (0/1)
- `minimal_valid.csv` — exactly 10 rows, 2 numeric columns (boundary minimum)
- `imbalanced_valid.csv` — 200 rows, minority class <10% (for SMOTE testing)

**Invalid CSV Files**:
- `file.xlsx` — wrong extension
- `too_large.csv` — > 50 MB (generated via padding)
- `too_few_rows.csv` — 5 rows only
- `no_numeric.csv` — all string columns
- `empty.csv` — zero bytes

**Built-in Datasets**:
- All 20 synthetic CSVs in `backend/data/` generated by `generate_synthetic.py`

## 7. Risk and Mitigation Table

| Risk ID | Risk Description | Likelihood | Impact | Mitigation |
|---------|-----------------|------------|--------|------------|
| R-001 | Large CSV upload causes server memory spike or timeout | Medium | High | 50 MB file size limit enforced in `constants.py`; streaming read via `file.read()` with size check before pandas parse |
| R-002 | Session expires during long testing session | Medium | Medium | Session TTL is 60 minutes (`SESSION_MAX_AGE_MINUTES = 60`); tests create fresh sessions; cleanup called on startup |
| R-003 | SMOTE fails for very small minority class (< k_neighbors samples) | High | Medium | `k_neighbors = min(5, max(1, min_class_size - 1))` dynamically set; fallback continues without SMOTE |
| R-004 | Column mapper gate bypass via direct API call with fabricated session_id | Low | High | `schema_ok` flag checked before prepare; no session bypasses are possible without a valid POST to /column-mapping |
| R-005 | Built-in dataset CSV missing from `backend/data/` | Low | High | `generate_synthetic.py` idempotently creates all 20 CSVs; Dockerfile copies data dir; checked in entry criteria |
| R-006 | Pandas `read_csv` misidentifies column types causing quality score errors | Medium | Low | Unit tests in `test_data_service.py` cover type detection logic; quality score computation has floor/ceil of 0-100 |
| R-007 | One-hot encoding of high-cardinality categorical column produces too many features | Medium | Medium | Warning surfaced to user via `imbalance_warning`; preparation still proceeds but UI advises reducing features |
| R-008 | Train/test split fails with stratify on single-sample class | Low | Medium | `try/except ValueError` in `preparation_service.py` retries without stratification |
| R-009 | Domain switch with active session corrupts in-progress pipeline | Medium | High | UI shows warning dialog before switching; frontend clears session state on confirmed switch |
| R-010 | Network failure between frontend and backend during Apply | Low | Medium | Frontend shows error banner with retry option; no partial state saved until full API response received |
